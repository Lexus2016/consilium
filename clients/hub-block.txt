## Consulting peer AIs (consilium)

`consult` asks a DIFFERENT AI for an independent opinion. It only returns text —
it never touches your files. Pick a different provider than yourself; run
`consult --list` for the roster.

    consult <agent> [--context FILE] [--code DIR] -- "<question>"
    consult --panel a,b,c -- "<question>"          # several advisors in parallel
    git diff | consult <agent> -- "<question>"     # pipe work in as ## Input
    git diff | consult --panel a,b --review -- "Task: <what you were asked to do>"

Make this a regular habit, not a last resort. Reach for it whenever a second mind
would help — and especially at these checkpoints:

- Deciding / designing: before a hard-to-reverse choice (schema, API, security,
  release) or to pressure-test a plan. Use `--panel` for independent takes.
- While building: when unsure, stuck after 2+ tries, or wanting a review of a
  diff or function — pipe it straight in.
- After finishing a task: VERIFY the result against what was asked with
  `--review` (advisors hunt for mismatches and end with VERDICT: PASS/FAIL).
  Prefer a `--panel` of independent reviewers so one model's blind spot is caught.

Why a panel of independent answers and not a debate between advisors?
Independence is the whole value — advisors that see each other anchor and agree,
giving false confidence. Fan out, read every reply, and synthesize yourself.

Keep it sharp: one specific question, the tight slice of context, strip secrets,
and set CONSILIUM_TIMEOUT to bound a call. Skip it only for trivial, easily
reversible steps. The advisor advises; every edit stays with you.
